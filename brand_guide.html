Brand System

Component guidance for a disciplined enterprise brand.

This guide defines how to use the repository as a visual system: restrained color, direct hierarchy, accessible interactions, and predictable deployment behavior.

At a Glance

One visual language, one operating model, one deploy story.
Editorial-enterprise tone with restrained red emphasis. Shared tokens for surfaces, typography, motion, and shell behavior. Governance, QA, and versioning treated as part of the product.

01 Principles

Reduce until the signal is unmistakable.

  • Use a small number of surfaces, rules, and emphasis states.
  • Prefer typography and spacing over decoration.
  • Use red only as a deliberate signal, not ambient color.

02 Tokens

One visual language across all components.

  • Shared surface, border, text, and accent tokens live in `theme.css`.
  • Component CSS should extend shared tokens, not re-invent the palette.
  • Cache-busted shared assets must remain version-aligned across every page.

03 Accessibility

Data must stay understandable without visual guesswork.

  • Standalone assets include hidden summary text for screen readers.
  • Focus states must remain visible on navigation and theme controls.
  • Reduced motion preferences should disable non-essential transitions.

04 Governance

Operational discipline is part of the brand.

  • Update `VERSIONING.md` rules together with any deploy-affecting asset changes.
  • Run the checklist in `QA.md` before trusting a Pages deployment.
  • Keep docs, gallery, and component inventory in sync.

Color Behavior

Monochrome base Black, off-white, graphite, and disciplined red accents.

Typography

System-first High-contrast size shifts, tight tracking, and clean uppercase metadata.

Surfaces

Reduced geometry Thin borders, shallow radii, minimal shadow, precise spacing.

Delivery

Static reliability Direct-open HTML files, GitHub Pages compatibility, explicit cache-busting.
Area Recommended Avoid
Color One accent family with neutral surfaces Multiple bright highlight colors in one component
Copy Short nouns, direct descriptions, measurable labels Marketing phrases or ambiguous placeholders
Structure Landmarks, summaries, predictable hierarchy Decorative wrappers with no semantic role
Change Control Shared token updates plus QA and version alignment One-off visual tweaks without deployment validation

05 Usage

Pair components according to business narrative.

  • Use `stat_card` with `sparkline_card` for KPI plus movement.
  • Use `bar_chart` with `bullet_chart` for mix plus target attainment.
  • Use `progress_bar` with `timeline_steps` for status plus phase context.

06 Contribution

System changes require documentation, not only code.

  • Update gallery, docs, and guide pages together.
  • Preserve direct-open previews and accessible summaries.
  • Use the manual QA checklist before trusting a release.
